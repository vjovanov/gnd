# Changelog

Records every notable change to `gnd`. Versions follow semver; the **latest release is inline** in this file, and **older releases live one-per-file under `docs/changelog/`** so a reader (human or agent) only loads the history they ask for. Each entry cites the FS/AS/G/DF IDs it touches, so the changelog is itself part of the conformant tree (`gnd .` validates the citations).

Schema-version bumps are called out explicitly: `gnd_config_version` ([§FS-config.5](functional-spec/FS-config.md#5-schema-versioning)) and the `AGENTS.md` init block version ([§FS-init.2](functional-spec/FS-init.md#2-outputs)). A bump to either is a breaking change for the consumer and must appear under **Changed** with a migration note.

## 1. Conventions

### 1.1 Sections per release

`Added`, `Changed`, `Deprecated`, `Removed`, `Fixed`, `Security`. Omit any section with no entries.

### 1.2 Schema version callouts

Any change to `gnd_config_version` or the `AGENTS.md` block version goes under **Changed** with the prefix `**Schema:**` and a one-line migration pointer.

### 1.3 Entry style

One bullet per change, present tense, leading with the affected ID. Example: `§FS-show: add --head mode for truncated output`.

### 1.4 Progressive discovery

Only **Unreleased** and the **most recent release** are inline. When a new release ships, the previous "latest" section is moved verbatim to `docs/changelog/<version>.md` and a one-line link is added under [§3 Older releases](#3-older-releases). The most recent release stays inline so the common reader and agent path — "what changed lately?" — is one file deep.

## 2. [0.1.0] — 2026-05-11

Release-candidate binary. The CLI covers the full subcommand surface the specs describe (§2.7) — including the opt-in grounding floor ([§FS-check.3.6](functional-spec/FS-check.md#36-ungrounded-source-file-opt-in)) and the coverage index ([§FS-cover](functional-spec/FS-cover.md#fs-cover-gnd-groups-citations-by-scanned-file)); the crates.io publish is still gated by [§RM-distribution-naming](roadmap.md#rm-distribution-naming-verify-package-names-before-first-publish), while the larger distribution arc — the `gnd-core`/`gnd-cli` workspace split, the npm/PyPI bindings, the optional `gnd-lsp` server, and `gnd check --watch` — plus the diff-aware co-change gate ([§RM-cochange-gate](roadmap.md#rm-cochange-gate-a-pre-commit--ci-recipe--no-impl-change-without-spec-and-test)) are tracked in `docs/roadmap.md`.

### 2.5 Renamed

- `gnd name` → `gnd id`; the spec `FS-name` → `§FS-id`. The subcommand that emits the next conflict-free ID for a new declaration is now invoked as `gnd id <KIND> "<title>"` — the verb names what it returns (an ID), matching the `ID=$(gnd id …)` idiom [§FS-init.2.2](functional-spec/FS-init.md#22-stdout--stderr) and the `AGENTS.md` scaffold already teach. No deprecation alias: `gnd name` is now an unknown subcommand — the `known commands:` list ([§FS-cli.4](functional-spec/FS-cli.md#4-errors-with-no-source-location)), the `gnd help` pages ([§FS-cli.3](functional-spec/FS-cli.md#3-cross-subcommand-flags)), the `--format`-accepting set ([§FS-errors.5](functional-spec/FS-errors.md#5-json-format)), and the bash/zsh/fish completion scripts all list `id`. The flag surface (`--width`, `--explain`, `--format text|json`), the slug/number derivation, and the `0`/`1`/`2` exit mapping are unchanged.
- `RM-ide-plugins` → `§RM-lsp`. Scope shrinks: ship the `gnd-lsp` crate and per-registry packages; document one-time editor setup snippets in the README; do not maintain per-editor wrappers.
- `gnd fmt --md-links` → `gnd fmt --cross-refs`; the config block `[fmt.md_links]` → `[fmt.cross_refs]` (keys `enabled` / `anchor_format` unchanged). A "cross-reference" is whatever construct the surrounding markup uses to point elsewhere — a Markdown inline link, an AsciiDoc `xref:`, a reStructuredText `:ref:` — so the flag is named for the concept, not for `md` + `links`. **Behaviour is unchanged: `--cross-refs` still emits exactly the Markdown inline-link form, and only in `.md` files** ([§FS-fmt.6](functional-spec/FS-fmt.md#6-cross-reference-emission-with---cross-refs)); language-specific cross-references stay out of scope for now (getting each renderer's anchor algorithm right is real work — [§DF-github-anchor-fidelity](decisions/functional/DF-github-anchor-fidelity.md#df-github-anchor-fidelity-the-github-anchor-profile-reproduces-github-slugger-exactly)). The general name and the `[fmt.cross_refs]` block ([§FS-config.3.7](functional-spec/FS-config.md#37-fmtcross_refs--cross-reference-emission)) leave room to add a second markup family later under the same flag and config — additive, no `gnd_config_version` bump. The decision and roadmap IDs that pre-date the rename keep their slugs (`§DF-md-link-emission`, `§DF-md-link-anchor-strategy`, `§DF-declaration-anchor`, `§RM-md-link-emission`) — IDs are stable handles ([§FS-non-goals](functional-spec/FS-non-goals.md#fs-non-goals-what-gnd-will-deliberately-not-do)); only their flag-name prose changed. `gnd config show`, `gnd fmt --help`, `templates/gnd.toml`, this repo's `.agents/gnd.toml`, `skills/gnd-init/SKILL.md`, the README, and the e2e cases (`fmt-md-links-*` → `fmt-cross-refs-*`) updated.
- Roadmap milestone IDs reslugged to fit this repo's slug-only `[id] format` now that `RM` is a checked kind: `RM-001-e2e-corpus` → `§RM-e2e-corpus`, `RM-002-show` → `§RM-show`, `RM-003-config` → `§RM-config`, `RM-004-marker-fmt` → `§RM-marker-fmt`, `RM-005-distribution` → `§RM-distribution`, `RM-006-lsp` → `§RM-lsp`, `RM-007-self-host` → `§RM-self-host`, `RM-008-core-cli-split` → `§RM-core-cli-split`, `RM-009-distribution-naming` → `§RM-distribution-naming`, `RM-010-md-link-emission` → `§RM-md-link-emission`, `RM-011-refs` → `§RM-refs`, `RM-012-watch` → `§RM-watch`. The shipped milestones (`§RM-e2e-corpus`, `§RM-show`, `§RM-config`, `§RM-marker-fmt`, `§RM-md-link-emission`, `§RM-refs`) keep a one-line declaration under `docs/roadmap.md` §"Shipped milestones" so their citations resolve; the detail stays in §2.7 below.

### 2.8 Fixed

- [§FS-show](functional-spec/FS-show.md#fs-show-gnd-reads-a-single-declaration-body-by-id): `gnd show <ID>.<section> --head` (and `--head --section S`) reported `section not found` for a section that exists — the `--head` short-circuit fired before the section was matched. It now prints the lead prose of the requested section; a missing section is still a `section not found` error. New e2e case `show-head-section-markdown` covers it.
- [§FS-fmt.2.3](functional-spec/FS-fmt.md#23-what-is-never-rewritten): `gnd fmt --marker` no longer rewrites ID-shaped file names inside Markdown link destinations, so a link like `[§FS-check](FS-check.md)` keeps its URL intact while the visible citation remains canonical.
- [§FS-cli.3](functional-spec/FS-cli.md#3-cross-subcommand-flags): `gnd check`, `show`, `refs`, `cover`, `fmt`, `list`, and `id` now reject a second path positional with `error: <subcommand> takes at most one path argument` (exit `2`) instead of silently using the last one and dropping the rest — for `check` the silent drop could turn `gnd check docs src` into a flood of misleading `unknown reference …` findings. `config` and `agent-setup-instructions` already enforced this; the rule is now uniform. New e2e cases `check-extra-path-arg` and `show-extra-path-arg`.
- [§FS-init](functional-spec/FS-init.md#fs-init-gnd-bootstraps-a-new-gnd-conformant-repo): `gnd init` without `--docs` no longer tells the user to "run `gnd check` — a freshly scaffolded tree is clean" as step 1 — without `docs/`/`e2e/` that run prints the empty-scan warning. The `next:` block now leads with scaffolding (`gnd init --docs` or creating the folders), then the clean `gnd check`. The empty-scan warning itself ([§FS-check.2.2](functional-spec/FS-check.md#22-empty-scan)) points at `gnd init --docs` rather than `gnd init`, which was the command the user had just run.
- [§FS-config](functional-spec/FS-config.md#fs-config-gnd-reads-a-toml-config-file-under-agents): `gnd config show` no longer prints `number_pattern` under a number-less `[id] format` (nor `slug_pattern` under a slug-less one) — the pattern only governs its placeholder, so emitting it implied a grammar that isn't in effect.
- [§FS-fmt.2.1](functional-spec/FS-fmt.md#21-trigger-to-marker) / [§FS-lsp.1.4](functional-spec/FS-lsp.md#14-live-trigger-transform): correct the `$$`-trigger description — it fires when the trigger is followed by a token matching the repo's `[id] format` ([§FS-config.3.2](functional-spec/FS-config.md#32-id--id-grammar)), not only by `<KIND>-<digit>`. The implementation already handled slug-only IDs (the `{kind}-{slug}` form `gnd` itself uses, where the trigger rewrites `$$FS-check` to the marker form); the spec text said otherwise and would have implied the trigger never fires in this repo.
- README: the `gnd cover --format json` example piped through `jq '.files[] | …'`, but the command emits NDJSON — one `{"path":…,"citations":[…]}` record per scanned file ([§FS-cover.3.2](functional-spec/FS-cover.md#32---format-json)) — so the example is now `jq -c 'select(…)'` with a note on the shape. Also drops the maintainer-only "run the Release registry names workflow before publish" line from the user-facing install section; that is owned by [§RM-distribution-naming](roadmap.md#rm-distribution-naming-verify-package-names-before-first-publish).
- [§FS-cli](functional-spec/FS-cli.md#fs-cli-gnds-command-line-surface-conventions): `gnd config <unknown>` now reports `error: unknown config command \`<word>\`` plus an `expected: config validate, config show` line — matching the "name what was given" shape the rest of the surface uses (`unknown kind …`, `unsupported shell …`) — and a leading-dash argument is reported as `unknown flag`. New e2e case `config-unknown-command`.
- `gnd --help`: the `config` line is capitalized like every other command line (`Validate or show …`, was `validate or show …`), and the `cover` line reads `Group the citation graph by scanned file.` instead of the self-redundant `Group every scanned file's citations by file.`. `cli-help` golden output updated.
- [§FS-fmt.6.2](functional-spec/FS-fmt.md#62-form) / [§DF-md-link-anchor-strategy.2.3](decisions/functional/DF-md-link-anchor-strategy.md#23-renderer-profiles): the `github` (and aliased `gitlab`) `anchor_format` profile now reproduces `github-slugger` byte-for-byte — disallowed characters are deleted in place and *each* remaining space becomes one `-`, with no run-collapsing and no trailing-`-` trim. Before this, `gnd fmt --cross-refs` emitted `#a-b` for `## A — B` and `#6-watch-mode-watch` for `` ## 6. Watch mode (`--watch`) ``, anchors that resolve nowhere on GitHub — the bug rode an erroneous "collapse consecutive `-`" clause in [§DF-md-link-anchor-strategy.2.3](decisions/functional/DF-md-link-anchor-strategy.md#23-renderer-profiles), now corrected by the new decision record [§DF-github-anchor-fidelity](decisions/functional/DF-github-anchor-fidelity.md#df-github-anchor-fidelity-the-github-anchor-profile-reproduces-github-slugger-exactly). `mkdocs` and `pandoc` are unchanged (Python-Markdown's TOC slugger genuinely collapses `[-\s]+`). New e2e case `fmt-cross-refs-profile-github` over a curated heading set; this repo's pre-commit hook gains `gnd fmt --cross-refs --write` and switches `lychee` to `--include-fragments` so anchor drift in `docs/` fails the commit.
- [§FS-fmt.6.2](functional-spec/FS-fmt.md#62-form): anchors derive from a heading's *rendered* text — an inline link `[text](url)` in a heading is reduced to `text` before slugifying, mirroring the renderer. This makes `gnd fmt --cross-refs` idempotent over a citation that lives in a section heading (`### 2.2 … supersede §FS-fmt.6.3` → `### 2.2 … supersede [§FS-fmt.6.3](…)`): before the fix, the second `fmt` pass slugified the raw source and mangled the anchor with the destination URL it had just inserted. The `fmt-cross-refs-profile-github` e2e case covers it.
- [§DF-github-anchor-fidelity](decisions/functional/DF-github-anchor-fidelity.md#df-github-anchor-fidelity-the-github-anchor-profile-reproduces-github-slugger-exactly): new decision record — the `github` anchor profile must match `github-slugger` exactly and operate on rendered heading text; records why the previous "collapse runs" description was wrong, why heading text must be reduced before slugging, and why `gnd check` never caught either (it validates the citation, not the `#fragment`).
- [§FS-fmt.6.2](functional-spec/FS-fmt.md#62-form) / [§DF-declaration-anchor](decisions/functional/DF-declaration-anchor.md#df-declaration-anchor-a-bare-id-markdown-link-points-at-the-declarations-heading-anchor): `gnd fmt --cross-refs` now gives a *bare-ID* citation an anchor too — `§G-<x>` wraps to `[§G-<x>](goals.md#g-x-the-title)` (the declaration's own heading), not a bare link to `goals.md` that drops the reader at the file top. Applies whenever the declaration's home is Markdown; a stub whose home is a source file (`# AS-x: [the scanner](src/x.rs)`) still gets a bare file link (a renderer won't jump inside a doc-comment), and `anchor_format = "none"` is unchanged. New decision record [§DF-declaration-anchor](decisions/functional/DF-declaration-anchor.md#df-declaration-anchor-a-bare-id-markdown-link-points-at-the-declarations-heading-anchor) amends [§FS-fmt.6.2](functional-spec/FS-fmt.md#62-form)'s old "no section → no anchor" rule; the `fmt-cross-refs-profile-github` e2e case gains a bare-ID anchor case, and running `gnd fmt --cross-refs --write` over this repo rewrites every bare-ID `.md` citation to carry the declaration anchor.
- [§FS-fmt.6.7](functional-spec/FS-fmt.md#67-configurability): the `mkdocs` `anchor_format` profile now keeps `-` in its slugs and collapses runs of `-`/whitespace to one separator, matching Python-Markdown's TOC slugger (`re.sub(r'[-\s]+', sep, …)`) — `# FS-1-x: Y` slugs to `#fs-1-x-y`, not `#fs1x-y`. Previously `-` was dropped, which only surfaced once [§DF-declaration-anchor](decisions/functional/DF-declaration-anchor.md#df-declaration-anchor-a-bare-id-markdown-link-points-at-the-declarations-heading-anchor) started emitting anchors derived from headings (declaration headings carry `-` in the ID).
- [§FS-fmt.6.2](functional-spec/FS-fmt.md#62-form): heading-text reduction (the "rendered text" rule) now also drops HTML-tag-shaped spans `<…>`, matching what a renderer does — `## RM-show: gnd show <ID>` slugs as `rm-show-gnd-show`, not `rm-show-gnd-show-id`. Surfaced by [§DF-declaration-anchor](decisions/functional/DF-declaration-anchor.md#df-declaration-anchor-a-bare-id-markdown-link-points-at-the-declarations-heading-anchor): the `<ID>` placeholders in the `RM-…` roadmap headings were producing anchors GitHub does not render. `lychee --include-fragments` over `docs/` (the pre-commit hook) is clean.
